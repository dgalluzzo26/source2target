Gainwell Source-to-Target Mapping Application

üéØ Overview

This Databricks Streamlit application provides an intelligent interface for mapping source database fields to target database fields using AI-powered semantic matching and vector search capabilities.

The application helps ETL engineers and data professionals efficiently map source table columns to target table columns by leveraging:

  • AI-Powered Mapping: Uses Databricks Foundation Models (Meta Llama 3.3 70B) to suggest intelligent field mappings
  • Vector Search: Employs Databricks Vector Search to find semantically similar target fields
  • Interactive UI: Streamlit-based interface for reviewing and selecting field mappings
  • Historical Context: Learns from previous mapping decisions to improve suggestions

üöÄ Getting Started

üìã Prerequisites

Required: Access to a Databricks workspace with SQL Warehouse, Vector Search endpoint, Foundation Model endpoint, and Unity Catalog access.
  1. Databricks Workspace: Access to a Databricks workspace with:
    • SQL Warehouse configured
    • Vector Search endpoint set up
    • Foundation Model endpoint available
    • Unity Catalog access
  2. Environment Variables:
    SERVICE_PRINCIPAL_CLIENT_ID=your_client_id SERVICE_PRINCIPAL_CLIENT_SECRET=your_client_secret DATABRICKS_HOST=your_workspace_url

‚öôÔ∏è Installation

  1. Install Databricks CLI:
    pip install databricks-cli
  2. Authenticate to Databricks:
    databricks configure
  3. Deploy the Application:
    # Deploy to development environment databricks bundle deploy --target dev # Deploy to production environment databricks bundle deploy --target prod
  4. Run the Application:
    databricks bundle run

  • Database Settings: Warehouse name, mapping table, semantic table

  • Vector Search Settings: Index name, endpoint name
  • UI Settings: App title, theme colors, sidebar preferences
  • Support Settings: Enterprise support portal URL

üîê Service Principal Permissions

The application service principal requires specific permissions to access Databricks resources. Ensure the following permissions are granted:

üìä Database Table Permissions

The service principal needs READ and WRITE permissions on these three core tables:

Required Table Access:
  • Mapping Table - Stores source-to-target field mappings
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE <catalog>.<schema>.<mapping_table> TO '<service_principal_id>';
  • Semantic Table - Contains target field metadata for vector search
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE <catalog>.<schema>.<semantic_table> TO '<service_principal_id>';
  • Previous Mappings Table - Tracks historical mapping decisions
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE <catalog>.<schema>.<previous_mappings_table> TO '<service_principal_id>';

üîç Vector Search Permissions

The service principal requires permissions to query and refresh the vector search index:

Vector Search Access:
  • Query Access - Execute semantic similarity searches
    GRANT EXECUTE ON FUNCTION vector_search TO '<service_principal_id>';
  • Index Refresh - Sync index when semantic data changes
    GRANT MODIFY ON VECTOR_SEARCH_INDEX <index_name> TO '<service_principal_id>';
  • Index Access - Read from vector search index
    GRANT USAGE ON VECTOR_SEARCH_INDEX <index_name> TO '<service_principal_id>';

üè≠ SQL Warehouse Permissions

The service principal needs access to execute queries on the SQL warehouse:

GRANT CAN_USE ON WAREHOUSE <warehouse_id> TO '<service_principal_id>';

üìñ Usage Guide

üöÄ Getting Started

  1. Access the Application: Navigate to the deployed Databricks App URL
  2. Configure Settings: Use the Configuration tab to set up database connections and AI endpoints
  3. Verify User Access: Check the user access filter to understand your data permissions
  4. View Unmapped Fields: The Field Mapping tab shows source fields that need target mappings
  5. Explore AI Suggestions: Click on any row to see AI-generated mapping suggestions with detailed reasoning
  6. Manage Mapped Fields: Use the Mapped Fields tab to view, search, and manage existing mappings
  7. Manage Semantic Data: Use the Semantic Management tab to maintain the vector search semantic table
  8. Save Decisions: Select and save your preferred mapping choices

üì§ Template Upload/Download System

The application includes a powerful template system for bulk uploading new mapping requests:

üì• Download CSV Template

  • Purpose: Get a properly formatted CSV template for bulk uploads
  • Template Structure: Contains all required source field columns:
    src_table_name, src_column_name, src_column_physical_name, src_nullable, src_physical_datatype, src_comments
  • Example Row: Template includes sample data showing the expected format
  • Location: Available in the Unmapped Fields tab under "Template Upload/Download"

üì§ Upload New Mapping Rows

  • File Format: CSV files with all required columns populated
  • Validation: Automatic validation ensures:
    • All required fields are present and non-empty
    • No duplicate rows for the current user
    • Proper data types and formats
  • Automatic Processing:
    • Generates query_text for AI model consumption
    • Sets source_owners to current user's email
    • Creates empty tgt_columns struct for future mapping
    • Prevents duplicate uploads for the same user
  • Feedback: Real-time validation with duplicate detection and error reporting
üí° Template Upload Workflow:
  1. Download the CSV template from the Unmapped Fields tab
  2. Fill in your source field information (all fields required)
  3. Upload the completed CSV file
  4. Review validation results and duplicate warnings
  5. Confirm upload to create new mapping requests
  6. New rows appear in Unmapped Fields for AI mapping suggestions

üîê User Access Control

The application implements row-level security based on source ownership:

üë§ Source Owners Filtering

  • Purpose: Ensures users only see and manage their assigned data
  • Implementation: Uses source_owners column containing comma-separated email lists
  • Access Rules:
    • Records with empty/null source_owners are visible to all users (global access)
    • Records containing the user's email in source_owners are visible to that user
    • Users can only update/delete records they have access to
  • Visual Feedback: User Access Filter info box shows current filtering status

üîç User Detection Methods

The application uses multiple methods to identify the current user:

  • Databricks runtime environment variables
  • Streamlit user context (experimental_user, context.user_info)
  • Databricks App headers (X-Forwarded-Email)
  • Manual user override for service principal deployments
  • Environment variables (DATABRICKS_USER_EMAIL)
üîí Security Note: All database operations include user filtering to prevent unauthorized access to data not owned by the current user.
